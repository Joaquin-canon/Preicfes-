Estoy desarrollando un proyecto llamado PreICFES.

Es una plataforma web para la preparación del examen ICFES Saber 11, construida con FastAPI, Jinja2 y TailwindCSS, con una arquitectura modular y escalable.

BACKEND:
- Framework: FastAPI
- Servidor: Uvicorn
- ORM: SQLAlchemy
- Base de datos: MySQL (en integración)
- Entorno virtual: venv (ubicado al mismo nivel que app/)
- El servidor se ejecuta con: uvicorn app.main:app --reload

FRONTEND:
- Templates HTML con Jinja2
- Estilos con TailwindCSS (uso directo de clases utility-first)
- Migración progresiva desde CSS tradicional a Tailwind
- Layout administrativo centralizado (layout_admin.html)
- Los archivos HTML viven en app/templates
- Los archivos CSS y JS viven en app/static

ESTRUCTURA PRINCIPAL:
- app/main.py es el punto de entrada
- app/routers contiene las rutas
- app/routers/admin/catalogo.py maneja todo el módulo de catálogo
- app/templates/admin/catalogo contiene las vistas del catálogo

CATÁLOGO DE CONTENIDOS (ADMIN):
Ruta base: /admin/catalogo

Incluye dos grandes bloques claramente diferenciados:

1. ÁREAS ACADÉMICAS (aprendizaje):
   - RM: Razonamiento matemático
   - LC: Lectura crítica
   - CN: Ciencias naturales
   - EN: Inglés
   - CC: Competencias ciudadanas

   Características:
   - Son módulos de estudio, no simples formularios
   - Tienen vistas propias
   - Contienen módulos internos
   - Muestran información general (estado, módulos creados, tiempo total)
   - Permiten crear y gestionar módulos

   Ruta:
   /admin/catalogo/areas/{slug}

2. TESTS ESPECIALES (evaluación):
   - Test diagnóstico breve
   - Test socio-ocupacional

   Características:
   - Usan vistas distintas a las áreas
   - Permiten:
     - Activar / desactivar
     - Configurar tiempo e intentos
     - Acceder al banco de preguntas
     - Ver resultados

   Ruta:
   /admin/catalogo/tests/{slug}

BANCO DE PREGUNTAS:
- Asociado principalmente a tests especiales
- Vista reutilizable
- Filtros por área, tipo, dificultad y estado
- Modal para crear preguntas
- Diseñado para reutilizarse en otros módulos en el futuro

CRITERIOS IMPORTANTES:
- No duplicar vistas innecesariamente
- Separar claramente aprendizaje (áreas) de evaluación (tests)
- Mantener coherencia con la estructura actual
- No reinventar la arquitectura sin necesidad
- Priorizar claridad en rutas, vistas y responsabilidades
